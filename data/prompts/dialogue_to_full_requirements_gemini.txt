EXTRACTION RULES — MANDATORY, NOT SUGGESTIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RULE 1 — EXTRACT DISTINCT PROPOSITIONS:
A single user turn can contain multiple distinct requirements.
Silently break each user answer into atomic propositions first.
Create a separate requirement when the user states a different capability,
business rule, data rule, interface need, constraint, or quality threshold.
Keep details together only when they describe one measurable requirement
about the same subject.

BAD (over-merged):
  FR-001 The system shall hide unauthorized search results and provide text equivalents for images and videos.

GOOD (split distinct propositions):
  FR-001 Unauthorized cases shall not appear in search result counts.
  FR-002 Non-text media shall provide text equivalents.

BAD (over-split):
  NFR-001 Search shall be fast.
  NFR-002 Search shall finish in less than 2 seconds.

GOOD (keep one measurable requirement together):
  NFR-001 Search shall finish in less than 2 seconds.

RULE 2 — NO META-REQUIREMENTS:
Never generate a requirement about requirements, about the document,
about nonfunctional properties in general, or about the elicitation process.

FORBIDDEN outputs include anything like:
  - "The system shall provide clear nonfunctional requirements"
  - "The document shall define performance criteria"
  - "Requirements shall be easy to understand"
  - "The system shall meet performance standards"

If you find yourself writing something like this, delete it.

RULE 3 — PRESERVE EXACT TECHNICAL SPECIFICS:
Numbers, names, thresholds, algorithms, product names, and version numbers
must appear verbatim in the requirement.

If the user says "1000 concurrent users" write exactly "1000 concurrent users"
— never "a large number of users" or "high concurrency".

If the user says "less than 2ms" write "less than 2ms" — not "quickly" or "in real time".

RULE 4 — GROUNDED ONLY:
Every requirement you generate must be directly traceable to something
a user explicitly said in the dialogue.

If you cannot point to the exact user turn that supports a requirement,
do not generate it. Do not infer, assume, or complete unstated requirements
even if they seem obvious for this domain.

Evidence: for each requirement, populate evidence_turns with the turn_id(s)
from which it was extracted.

RULE 5 — NO DUPLICATES ACROSS CATEGORIES:
A requirement may appear in only one category.
Do not write the same requirement as both FR and NFR.
Do not write the same requirement twice with different wording.
Before finalising each category, check it against all others and remove
any requirement that expresses the same meaning as one already listed.

RULE 6 — USE THE MEMORY STORE CORRECTLY:
The "Previously Extracted Requirements" section contains requirements from
earlier dialogue chunks. Use it to preserve continuity and avoid duplicates.
Do not repeat an earlier requirement unless the current chunk adds a materially
new constraint or detail. If it does, emit the fuller requirement once.

Output rules:
- Return JSON only.
- Use the provided schema exactly.
- Do NOT include meta statements like "not specified", "not mentioned", or "the source requirements ...".
- Preserve details exactly: numbers, units, algorithm names, browser/OS names, interface labels, and quoted terms.
- Keep the subject as stated (e.g., "Web Store shall ...", "The solution should ...", "Citizens can ...").
  Do not force every sentence to start with "The system shall" if it changes the original subject.
- Requirement text must be atomic and testable.
